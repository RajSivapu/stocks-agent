# Project 1 — v2.1 design: intraday discovery + the owner's learning loop

**Date:** 2026-06-18
**Status:** approved (brainstorming) — pending implementation plan
**Builds on:** `2026-06-17-project1-v2-autonomous-daily-operator-design.md` (v2 "Autonomous Daily Operator")
**Guardrail (unchanged, non-negotiable):** suggestion-only, read-only. The agent NEVER places, modifies,
or cancels a trade or a stop order. It only *suggests*; Rajrupesh executes manually on Robinhood.

## Why

v2 shipped a full discovery → historical-check → alert-or-waitlist → grade → learn loop, but it all runs
at the 06:30 pre-market brief; the intraday runs only babysit open zones + holdings. The owner wants
(1) mid-day opportunity hunting, (2) a place to track *his own* hypotheses separately from the agent's,
(3) every buy to carry a clear exit as well as an entry, and (4) his holdings' stops to trail up as a
stock rises. All four are **additive** to the v2 architecture (Postgres + `lib/` + skills). The agent's
suggestion / grading / radar / holdings core logic is **not** rebuilt.

## Scope — four features

### Feature 1 — Intraday becomes a bounded opportunity hunter (+ self-validating waitlist)

Replace the two stripped-down intraday runs with **one** mid-day intraday run (default ~12:00 CT) that
does BOTH a position monitor and a *bounded* discovery pass.

- **Part A — Monitor (runs first, cheap):** open entry-zones (`lib.db.get_open_suggestions`) — has price
  entered the zone? hit invalidation? Holdings (`lib.db.get_holdings`) — hit stop/invalidation? These are
  the only *position* alerts.
- **Part B — Bounded discovery (new):**
  1. **Refresh the waitlist** — re-check radar names (≤ `radar.max_size` = 15): live quote, update
     `last_seen` / `days_relevant`, note today's behavior.
  2. **Catch fresh catalysts** — one cheap movers / most-active + breaking-news screener pull (same
     endpoints the morning scan uses), apply `market_scan.universe_filters` quality floor.
  3. **Evaluate the promising few** — top ~3 candidates only (radar names acting well + new movers with a
     real catalyst): historical-behavior check (`lib.db.get_observations` / self-seed) + a **compact**
     structured deliberation + the existing confidence/risk gate.
  4. **Decide each:**
     - Clears the buy-gate (Medium+ conviction, in/near a sensible entry zone, dry powder available) →
       **🔔 ALERT** as a buy idea (entry zone + target + stop + valid-until) and log a Buy suggestion.
     - Promising but not yet a buy → add/refresh on the **radar (waitlist)** AND log a **Watch suggestion
       with today's price** (so the existing grading pass scores the hunch later). No Telegram.
     - Not worth it → drop / let it prune.
- **Notification policy:** intraday pings Telegram ONLY for (a) a new buy that cleared the gate, (b) an
  open entry-zone trigger, or (c) a holding stop/invalidation hit. Everything else is silent DB writes.
  Never an "all-clear" message.
- **Cost guardrail (new hard cap for the intraday run):** ≤ ~25 data API calls; deep-analyze ≤ 3 names;
  **compact depth only** (full multi-round debates stay in the 06:30 brief). Well under the morning ≤70.

**Self-validation ("was I right?") — reuses existing machinery:** promising finds are logged as Watch
suggestions with a price, so the existing grading pass grades them at 5/21/63 days and tempers confidence
by bucket. The radar `days_relevant` tracks sustained relevance and auto-promotes to the watchlist after 5
relevant days. "Set amount" = the radar cap of 15 names; money to act on a promotion = the existing
growth/spec dry-powder budget.

### Feature 2 — The owner's paper-watch table (core + you-vs-agent)

A place for the owner to track *his own* hypotheses, fully separate from the agent's radar and from real
holdings. Three distinct concepts, never mixed: `radar` = agent's hunches; `holdings` = real money;
`paper_watches` (new) = owner's hypothetical bets, for learning.

- **New table `paper_watches`:** `id, ticker, created (date), entry_ref_price (numeric), target_price
  (numeric, nullable), hypothetical_amount (numeric, nullable), thesis (text), horizon (text:
  "days"/"weeks"/"months"/open), status (text: active|closed), closed_date (date, nullable), close_price
  (numeric, nullable), agent_view_at_open (text, nullable — the you-vs-agent snapshot), agent_score_at_open
  (int, nullable).`
- **Add/close in plain English** via a new small skill `paper-watch` (mirrors `reconcile-trade`'s
  parse-and-write pattern, but writes ONLY `paper_watches` — never holdings/transactions):
  - *"Paper-watch NVDA from $200, thesis: GPU cycle has legs, track a month"* → insert a row; snapshot the
    agent's current stance on NVDA into `agent_view_at_open` / `agent_score_at_open`.
  - *"Close my NVDA paper watch"* → set `status=closed`, `closed_date`, `close_price`, report final
    hypothetical P&L.
- **Mark-to-market** in the post-market run (prices already fetched): for each active watch, fetch current
  price → hypothetical return % (and $ if `hypothetical_amount` set). If `target_price` is set and hit,
  auto-close as a hypothetical win.
- **Report — `🧪 Your paper watches` block in every daily-status brief** (horizon-aware: short-horizon
  watches always surface AND are picked up by the intraday monitor). Shows mark-to-market + the you-vs-agent
  line: `NVDA — flagged $200, now $214, +7.0% (+$7 on $100), 9 days · agent had it Watch → you both called it`.
- **Optional later:** a monthly `🧪 Paper-watch scorecard` alongside the agent's own scorecard. (Not in
  the first build; data model supports adding it.)

### Feature 3 — Every buy carries entry + exit/target + stop

The `suggestions` table already has `target` and `stop` columns (no migration). The change is behavioral:

- The skill must **always compute and populate `target` (take-profit) and `stop`** for every Buy
  suggestion, alongside the existing entry zone + valid-until + invalidation.
- The brief shows all of it on a buy line:
  `Buy zone: $198–$203 · Target: $235 · Stop: $188 · valid until Jun 25`.
- **Late-look safety:** if live price has already run past the entry zone, the alert says so plainly
  ("price has moved above the buy zone — wait for a pullback or re-check") rather than implying a fill that
  no longer exists. The intraday re-check enforces the same.
- Strategy note (sets expectations): Core DCA + the growth pick are buy-and-hold; the agent does NOT churn
  in/out over days. Shorter horizons occur mainly in the speculative/opportunistic bucket — exactly where
  the explicit target + stop matter most.

### Feature 4 — Trailing stop on holdings (advisory)

Once the owner holds a name, the agent reassesses the stop each run and suggests **raising** it as the
stock rises — locking in gains. Suggestion-only: the owner moves the stop on Robinhood himself.

- **Holdings migration:** add `stop (numeric)`, `target (numeric)`, `high_water_price (numeric)` columns.
- **On buy** (`reconcile-trade`): capture the `stop` + `target` from the recommendation the owner acted on
  and store them on the holding. If no suggestion is referenced, infer from the most recent suggestion for
  that ticker, else set a sensible default stop (a technical level / set % below entry) and note the
  assumption.
- **Each run, per holding:** update `high_water_price`; recompute a recommended stop from a
  **configurable trailing rule** (default, beginner-friendly: once up ≥ `breakeven_trigger_pct` (≈8%) move
  the stop to at least breakeven (`avg_cost`); then trail to the higher of a technical level (recent swing
  low / SMA50) and `price × (1 − trail_pct)`). **Cardinal rule: the stop only ratchets UP, never down.**
  Persist the new recommended `stop`.
- **Where it shows (split by urgency):**
  - **Raise-your-stop suggestion → daily-status brief** holdings section (post-market recomputes overnight
    so the morning brief is current): *"AAPL +6% — consider raising your stop $215 → $230 (now locks in a
    gain)."* Shown only when the recommended stop > the stored stop.
  - **Stop-hit → intraday `⚡ Market Alert`** (the urgent downside event the intraday already watches).
- The owner can sync his actual stop with *"moved my AAPL stop to 230"* (handled by `reconcile-trade`).

## Cadence decision (recorded)

Move from the v2 four-run plan to **three runs/day** to keep Pro budget in check while the intraday does
more useful work:

| Run | Time (CT) | Weight | Behavior |
|---|---|---|---|
| pre-market | 06:30 | heavy | Full brief (unchanged; monthly-plan on 1st weekday, else daily-status). |
| intraday | ~12:00 (default; confirm at routine creation) | light–medium | Feature 1 (monitor + bounded discovery). |
| post-market | 15:10 | medium | Analysis/learning + paper-watch mark-to-market + trailing-stop recompute. |

Honest tradeoff: the bounded intraday is pricier than the old monitor-only run, so net Pro usage may land
near the original four-light-run plan even at three runs — but it does real discovery instead of passive
watching. Measure ~2 weeks (per `routines/README.md`) and tune.
